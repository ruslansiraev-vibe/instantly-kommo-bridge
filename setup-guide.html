Setup Guide

Instantly → Kommo: Direct Integration

Webhook-based bridge without third-party intermediaries. Solves the «matryoshka» problem.

Date: 4 April 2026
Priority: High
Stack: Python + FastAPI
Status: Ready to deploy
Problem

Why this script exists

We use Instantly for cold email outreach and Kommo as our CRM. Previously, replies from Instantly were forwarded to a corporate email connected to Kommo. This created a cascade of problems we call the «matryoshka problem».

The matryoshka problem

When Instantly forwards every email to the inbox connected to Kommo, the CRM has no way to distinguish between useful replies and noise. The result:

  • Warm-up emails create junk deals — Instantly sends warm-up messages to maintain sender reputation, and every one of them ends up as a new deal in Kommo
  • Negative replies flood the CRM — “not interested”, “unsubscribe me”, out-of-office auto-replies all create deals that sales reps waste time closing
  • Duplicate contacts everywhere — there is no check whether a contact already exists, so every new reply creates a brand new contact record
  • Duplicate deals — every reply from the same lead creates a new deal instead of adding a note to the existing one, making it impossible to track conversation history
  • Lost context — the conversation thread is scattered across multiple deals, so when a sales rep picks up a lead they have no idea what was discussed before

Scale of the problem

With cold outreach to large databases, each email account generates ~60 new contacts per month. Multiply by the number of sending accounts, and the CRM fills up with hundreds of garbage records that obscure real opportunities.

Previous solution: Make.com

What we tried

The first attempt to solve this was a Make.com scenario (Instantly → Make.com → Kommo) that filtered replies and deduplicated contacts. While it worked, it introduced its own issues:

  • Third-party dependency — Make.com is another paid service ($29+/mo) that can go down, change pricing, or hit operation limits
  • Operation limits — Make.com charges per operation; high-volume outreach burns through quotas quickly
  • Limited control — debugging and customizing filter logic in Make.com's visual editor is cumbersome compared to code
  • Latency — an extra hop through a third-party service adds delay

This solution: direct bridge

What this script does

A lightweight Python microservice that connects Instantly and Kommo directly, without any intermediary. It:

  1. Receives webhooks from Instantly when leads reply or are classified as interested
  2. Filters — only positive replies pass through (Interested, Meeting Booked, Meeting Completed, Won). Warm-up, negative, OOO, and auto-replies are silently dropped
  3. Deduplicates contacts — searches Kommo by email before creating; one email = one contact
  4. Deduplicates deals — searches for an active deal linked to the contact; one contact = one active deal
  5. Preserves context — every new reply is added as a note inside the existing deal, building a full conversation history
  6. Prevents duplicate processing — SQLite store tracks processed email IDs, so webhook retries don't create duplicates
Result: Clean CRM with only qualified leads, no duplicates, full conversation history in each deal, zero third-party dependencies, and complete control over the integration logic.
Architecture

How it works

Instantly sends webhook events when leads reply. Our Python microservice receives them, filters out junk (warm-up, negative, auto-reply), deduplicates contacts and deals, and pushes clean data into Kommo via REST API. No Make.com, no Zapier, no intermediaries.

Instantly Your Server Kommo CRM ──────────── ────────────── ────────── reply_received ──webhook──> FastAPI server lead_interested ──webhook──> | lead_meeting_booked ─────> | v Filter: only positive replies Dedup: SQLite (email_id) | v Kommo API: search contact by email ┌──────┴──────┐ found not found | | | create contact | | v v Kommo API: search active lead ┌──────┴──────┐ found not found | | | create lead | | v v Kommo API: add note with reply text

Step 1

Kommo: create integration & token

1

Create a private integration

  1. Kommo → Settings → Integrations
  2. Click + Create integration (private)
  3. Name: Instantly Bridge
  4. Permissions: contacts, leads, notes — read + write
2

Generate a long-lived token

  1. Open the integration → Keys and scopes tab
  2. Click Generate long-lived token
  3. Set expiration: 1 year
  4. Copy the token immediately — it will not be shown again
Why long-lived token? Unlike OAuth 2.0 (access token expires every 24h, refresh token every 3 months), a long-lived token works like a simple API key. No refresh logic needed. Just regenerate once a year.
3

Create a pipeline for Instantly leads

Kommo → Settings → Pipelines. Create new pipeline:

StageColorDescription
New ReplyBlueJust received a reply from lead
In ConversationYellowActive dialogue
Meeting BookedGreenCall scheduled
Meeting DoneGreenCall completed
PartnershipGreenLead agreed to work
RejectedRedLead declined
4

Get Pipeline ID and Status ID

  1. Open Kommo in browser
  2. Press F12 (DevTools) → Network tab
  3. Go to any lead page or pipeline settings
  4. Find request to /api/v4/leads/pipelines
  5. In the response JSON, find your pipeline's id and first stage status_id
// Example response (find your pipeline):
{
  "id": 7654321,          // ← this is KOMMO_PIPELINE_ID
  "name": "Instantly Inbound",
  "_embedded": {
    "statuses": [
      {
        "id": 54321098,    // ← this is KOMMO_PIPELINE_STATUS_ID (first stage)
        "name": "New Reply"
      }
    ]
  }
}
Step 2

Instantly: API key

1

Generate API v2 key

  1. Instantly → Settings → Integrations → API
  2. Click Create API Key
  3. Scopes: minimum emails:read, leads:read. Best: all:read
  4. Copy the key — you'll need it in .env
2

Generate a webhook secret

Create a random secret for webhook signature verification:

openssl rand -hex 32
# Example output: a3f8b2c1d4e5f6...  ← save this as INSTANTLY_WEBHOOK_SECRET

Launch the server

1

Start locally

# Start the server

# Verify

2

Make the server publicly accessible

Instantly needs to reach your server over the internet. Choose one:

OptionCommandBest for
ngrok (testing) ngrok http 8000 Quick test, free, temporary URL
Cloudflare Tunnel cloudflared tunnel --url http://localhost:8000 Free, stable, no port opening
VPS + nginx Deploy on DigitalOcean / Hetzner Production: SSL, uptime, reliability
For production — use a VPS with nginx reverse proxy, SSL certificate (Let's Encrypt), and systemd service for auto-restart. ngrok is only for testing.
3

Production setup (VPS)

# nginx config (/etc/nginx/sites-available/instantly-bridge)
server {
    listen 443 ssl;
    server_name bridge.yourdomain.com;

    ssl_certificate     /etc/letsencrypt/live/bridge.yourdomain.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/bridge.yourdomain.com/privkey.pem;

    location / {
        proxy_pass http://127.0.0.1:8000;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }
}

# systemd service (/etc/systemd/system/instantly-bridge.service)
[Unit]
Description=Instantly-Kommo Bridge
After=network.target

[Service]
User=www-data
WorkingDirectory=/opt/instantly-kommo-bridge
EnvironmentFile=/opt/instantly-kommo-bridge/.env
ExecStart=/opt/instantly-kommo-bridge/venv/bin/uvicorn server:app --host 127.0.0.1 --port 8000
Restart=always
RestartSec=5

[Install]
WantedBy=multi-user.target

Step 6

Testing

Test checklist

  1. Test 1: New positive reply
    Send a test reply in Instantly (or use register_webhooks.py test endpoint).
    Expected: Contact created in Kommo + Lead created + Note with reply text added.
  2. Test 2: Repeat reply from same lead
    Send another reply from the same email address.
    Expected: NO new contact, NO new lead. Note added to EXISTING lead.
  3. Test 3: Negative reply
    Lead replies with “not interested”.
    Expected: Nothing reaches Kommo (filtered out).
  4. Test 4: Auto-reply / OOO
    Expected: Nothing reaches Kommo.
  5. Test 5: Duplicate webhook
    Same email_id arrives twice (Instantly retry).
    Expected: Processed only once (SQLite dedup).
Check server logs: tail -f /var/log/syslog | grep instantly-bridge or watch uvicorn output directly.
Step 7

Disable old email forwarding

Critical: remove the old integration

After all tests pass, disable email forwarding from Instantly to corporate email linked to Kommo. This is the root cause of the «matryoshka» problem.

  1. Instantly → Settings → Email Accounts
  2. Remove or disable the forwarding rule to *@yourcompany.kommo.com
  3. Verify: send a test email — it should NOT appear in Kommo inbox, only via the bridge

Reference

API documentation links

Instantly API v2

  • Docs: developer.instantly.ai
  • Webhooks: POST /api/v2/webhooks — create
  • Emails: GET /api/v2/emails — list replies (fallback polling)
  • Auth: Bearer token in Authorization header
  • Rate limits: 20 req/min (email listing)

Kommo API v4

  • Docs: developers.kommo.com
  • Contacts: GET/POST /api/v4/contacts
  • Leads: GET/POST /api/v4/leads
  • Notes: POST /api/v4/leads/{id}/notes
  • Auth: Long-lived token or OAuth 2.0
  • Long-lived token: valid 1 day – 5 years, no refresh needed
